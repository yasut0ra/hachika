# プロダクトレビュー（2026-03-21）

## 対象
- 対象リポジトリ: `hachika`（Node.js + TypeScript の実験的 CLI）
- レビュー観点: 価値提案、体験、実装成熟度、運用容易性、今後の優先順位

## 総評（TL;DR）
Hachika は「ユーザー最適化ツール」ではなく「内的駆動から自己目的を形成する存在」を目指す点で、
コンセプトが非常に明確で差別化も強いです。現時点の v0 としては、
drive/state/trace/purpose/identity まで一貫したモデルを実装し、
さらに artifact の外部化まで到達しているため、**研究プロトタイプとしての完成度は高い**です。

一方で、プロダクトとして広く試用を促進するには、
「初回体験の分かりやすさ」「観測性（なぜその返答になったか）」「評価指標の定義」がまだ不足しています。

## スコアリング（5点満点）
- ビジョンの独自性: **5.0**
- 実装の一貫性（コンセプト→コード）: **4.5**
- 体験の分かりやすさ（初見ユーザー）: **3.0**
- 運用・開発のしやすさ: **3.5**
- 検証可能性（実験としての測定容易性）: **3.0**
- 総合: **3.8 / 5.0**

## 良い点
1. **コンセプトと実装が接続されている**
   - 原初的駆動、身体状態、記憶、purpose、identity、initiative が分断されずに連携している。
2. **テスト密度が高い**
   - 単体だけでなくシナリオ系まで含み、回帰しやすい設計になっている。
3. **artifact の materialize 設計が強い**
   - 「会話外に痕跡を残す」という思想が、実ファイル出力まで実装されていて検証可能。
4. **LLM 非依存のフォールバック設計**
   - input interpreter/reply generator が失敗しても local rule で継続でき、実験の安定性が高い。

## 課題
1. **初回オンボーディングの摩擦**
   - 研究的概念は強いが、初見ユーザーが「何を入力すれば面白くなるか」を掴みにくい。
2. **評価軸の明文化不足**
   - 「生命らしさ」「自己志向性」の改善をどう測るか、定量・定性の指標が不足。
3. **観測性のUI不足（CLI中心）**
   - 内部状態は豊富だが、セッション中の変化を直感的に追いにくい。
4. **開発体験の小さなつまずき**
   - テストスクリプトの glob 指定が環境により失敗しうるなど、試行時の微小な摩擦がある。

## 直近4週間の推奨アクション
1. **DX整備（最優先）**
   - `npm test` / `npm run build` / `npm run dev` が README の手順通り確実に通る状態を維持。
   - 代表シナリオをワンコマンドで再生できる `npm run scenario` の追加を検討。
2. **評価指標の導入**
   - 例: 継続率、再訪時の一貫性、衝突後の関係修復率、topic継承率。
   - `metrics.md` を作り、各リリースで比較可能にする。
3. **体験テンプレートの提供**
   - 「関係形成」「共同作業」「境界衝突」など、5〜10個の会話プロンプト例を README に追加。
4. **可視化の最小導入**
   - セッションごとの state/drive 変化を時系列で表示する軽量ビューア（CLI出力拡張でも可）。

## 中長期の方向性
- 目的形成の品質を上げるため、`active purpose` の評価関数を強化（達成/放棄だけでなく満足度を導入）。
- artifact 再利用の精度改善（検索・参照優先度・重複統合）。
- 倫理的/安全的ガードレールの明文化（研究利用時の注意事項、境界条件、ログポリシー）。

## hachika-v1 計画案（提案）

### v1のゴール
- 実験プロトタイプ（v0）から、**再現可能に評価できる実験プロダクト**へ進める。
- 「自己志向的ふるまい」の観察を、会話ログだけでなく指標・可視化で検証可能にする。

### スコープ案（Must / Should / Could）

#### Must（v1で必須）
1. **評価基盤の実装**
   - セッション単位で `continuity / relation / curiosity / boundary conflict` などの変化量を記録。
   - 最低限の比較レポート（run間比較）を出力。
2. **観測性の強化**
   - 返答ごとに「主要 motive / active purpose / 参照 trace」を明示する lightweight diagnostics を標準化。
3. **オンボーディング改善**
   - 初回ユーザー向けに対話テンプレート（5〜10本）を同梱し、最初の10ターンで体験価値が出るようにする。
4. **運用安定化**
   - CIで `build + test + lint(導入時)` を通す。
   - 主要設定（env）のバリデーションを追加。

#### Should（余力があればv1に入れる）
1. **artifact再利用の改善**
   - 既存 trace / archived trace の再活用優先度を改善（重複抑制、reopen判断の精度向上）。
2. **目的遷移の品質向上**
   - active purpose の達成・放棄だけでなく、満足度や未練の状態を持たせる。
3. **評価シナリオ固定化**
   - 回帰検証用に標準シナリオ（関係形成・衝突・修復・再開）を固定し、毎リリース比較。

#### Could（v1.1以降候補）
1. 簡易Webビューア（state/trace推移の可視化）
2. マルチセッション横断のアイデンティティ要約
3. 安全ポリシー文書と研究利用ガイド整備

### マイルストーン（8週間想定）
- **W1-2:** 指標定義・ログ項目確定・最小ダッシュボード（CLI出力）
- **W3-4:** diagnostics標準化・オンボーディングテンプレート追加
- **W5-6:** purpose遷移とartifact再利用の改善
- **W7-8:** 評価シナリオ回帰を回して調整、v1ドキュメント化

### v1 完了条件（Definition of Done）
- 同一シナリオで、主要指標の比較結果をリリースノートに添付できる。
- 初回体験テンプレートで、ユーザーが3種類以上の「自己志向的ふるまい」を再現観察できる。
- CIで build/test が安定し、既知の回 regressions が管理されている。
